## 三、能否「跟著 v1.5.0 把缺的補完」？

**可以，但要分清楚哪些是廉價、哪些是無底洞**：

### 3.1 廉價部分

- 工具收斂為 action router：純機械重構，1-2 週可完成。但需注意
  **token 改善是行銷數字**，實際取決於 client 是否會把所有 action 與其
  conditional schema 全載入 prompt。
- 面板 UI 重做：1-2 天，無風險。
- 接口/文件補完：邊做邊補。

### 3.2 中等成本

- 事件綁定：要研究 `cc.EventHandler` 的序列化格式與 Editor 設定
  channel，估 2-3 天加實機測試。

### 3.3 高成本但非無底洞（2026-04-30 重評）

- **預製體完整修復**：原列「無底洞」。重新評估後降級為**中等成本**——
  關鍵在於 fallback ladder（`prefab-tools.ts:360-364` 的 3 個連接 channel
  與 `:621-623` 的 3 個 apply channel）多數是憑空猜的，依
  CLAUDE.md 原則「用 `@cocos/creator-types` 驗證頻道而非 try-catch 階梯」
  就能逐一確認哪個是真存在的。
  - 拆分後 ladder 清理約 1 天（純驗證 + 換成單一 channel）。
  - 「100% 完整」（含 fileId / `__id__` 全鏈路）仍需實機測試，視
    結果再加碼。
  - 排入 P1 新增的 T-P1-6（見 `roadmap/02-architecture.md`）。

## 四、結論（2026-04-30）

**仍不建議把「對齊 v1.5.0」當主目標**，但 v1.5.0 承諾的部分項目可選擇性
拿回來。理由：

1. v1.5.0 重頭戲（預製體 100%、事件綁定）是高/中成本項目，但**並非全部
   無底洞**——預製體 ladder 清理屬可控範圍，事件綁定屬一般功能新增。
2. v1.5.0 的優先序與「修穩當前 fork」的優先序衝突——前者要先做大重構
   再修細節，後者應該先把硬傷（硬編碼路徑、JSON 修補、log 噪音）清掉。
3. v1.5.0 的「token -50%」未驗證。盲跟可能反而退化（仍維持 P2 條件式
   執行）。

**修訂路線**（見 `roadmap/README.md`）：
- P0（done）→ P1 架構債（新增 T-P1-6 預製體 channel 驗證）→
  P2 工具收斂（條件式）→ **P4 v1.5.0 部分對齊**（事件綁定 + 面板 UI）→
  P3 協定能力擴展。
- P3 與 P4 互不相依，順序視當下需求。
