## 三、能否「跟著 v1.5.0 把缺的補完」？

**可以，但要分清楚哪些是廉價、哪些是無底洞**：

### 3.1 廉價部分

- 工具收斂為 action router：純機械重構，1-2 週可完成。但需注意
  **token 改善是行銷數字**，實際取決於 client 是否會把所有 action 與其
  conditional schema 全載入 prompt。
- 面板 UI 重做：1-2 天，無風險。
- 接口/文件補完：邊做邊補。

### 3.2 中等成本

- 事件綁定：要研究 `cc.EventHandler` 的序列化格式與 Editor 設定
  channel，估 2-3 天加實機測試。

### 3.3 無底洞

- **預製體完整修復**：依賴 Cocos Editor 內部 `Editor.Message` 頻道，官方
  文件不全。原作者 v1.4.0 自稱修完，程式碼仍是 ladder，代表這塊
  **沒人真的「修完」過**。要做需要：
  1. 實機 Cocos Creator + 測試專案。
  2. 對 3.8.6（與目標版本）逐一驗證 channel 是否存在。
  3. 處理 `_prefab` 內部結構序列化（fileId 對應、`__id__` 引用）。

## 四、結論

**不建議把「對齊 v1.5.0」當主目標**。理由：

1. v1.5.0 重頭戲（預製體 100%、事件綁定）是高風險高成本項目。
2. v1.5.0 的優先序與「修穩當前 fork」的優先序衝突——前者要先做大重構
   再修細節，後者應該先把硬傷（硬編碼路徑、JSON 修補、log 噪音）清掉。
3. v1.5.0 的「token -50%」未驗證。盲跟可能反而退化。

**建議路線**：以 `roadmap/01-baseline-fixes.md` 為起點，先把 P0 硬傷修掉、
P1 架構債（換 SDK、去重複實例化）清掉，再評估是否值得把工具收斂成 v1.5.0
的 action router 形式。
