# ADR 0001: 不追原作者 v1.5.0 spec

**Date**: 2026-04-30
**Status**: Accepted
**Decider**: shang（fork owner）

## Context

Fork 自 `arnie1128/cocos-mcp-server` 於 v1.4.0。原作者在 README 宣告 v1.5.0
已於 Cocos Store 上架（2024-07-29），但 GitHub 主線並未推送該版本，且自
2025-07-26 後上游無進一步 commit。

v1.5.0 README 承諾的主要改動為：

1. 150+ 工具收斂為 50 個 action router 工具（聲稱 token 消耗 -50%）
2. 預製體建立／實例化／引用「100% 對齊官方格式」
3. 補完事件綁定（onClick 等）
4. 接口優化、面板 UI 簡潔化
5. 整體架構效能提升

接手此 fork 後第一個策略決定：**追上 v1.5.0** 還是**自走優化路線**？

## Decision

**不追 v1.5.0 spec。**改走漸進式架構債清理 + 條件式工具收斂的自走路線
（見 `docs/roadmap/`）。

## Consequences

### Positive

- 不被原作者「100% 修完預製體」的不可能任務綁架——該 spec 在現存程式碼上
  仍是一堆 try-catch fallback ladder（`prefab-tools.ts:359-666`），代表
  原作者也沒真的「修完」。
- 可優先處理 v1.5.0 沒列入但更迫切的硬傷（硬編碼路徑、JSON 修補
  regex、log 噪音）。
- 可引入 v1.5.0 沒提到但 CP 值更高的優化：`@modelcontextprotocol/sdk`、
  zod schema、Resources/Prompts/Notifications。
- 工具收斂改為條件式——若 P1 完成後 token 量測證明已夠精簡，就不必收斂；
  反之才執行。避免做白工。

### Negative

- Cocos Store 使用者（裝商城版本者）若期待此 fork 對齊 v1.5.0 介面，會
  失望。
- 需要自己撰寫文件解釋差異，不能直接導向上遊 README。
- 若上游突然復活並推 v1.5.0 程式碼，merge 成本會比直接跟上更高。

### Neutral

- 工具命名可能與 v1.5.0 不同，發 release 時需明確標示為 fork 而非
  drop-in replacement。

## Alternatives Considered

### Option A: 完全照 v1.5.0 spec 重做

**為什麼沒選**：

- 預製體 100% 完整修復是無底洞，依賴 Cocos Editor 內部未文件化的
  `Editor.Message` 頻道，需要實機 + 大量試誤。
- Token -50% 是行銷數字，未經量測；盲跟可能反而退化（conditional
  schema 的字數有時比原本的扁平 schema 還多）。
- 與「先修硬傷再加新功能」的工程順序衝突——v1.5.0 把架構重寫與細節修補
  綁在一起，不易拆解。

### Option B: 完全重寫（不基於 v1.4.0）

**為什麼沒選**：

- 重寫成本 > 重構成本，且會丟失原作者已驗證的工具實作（170 個工具，多數
  仍可用）。
- 重寫需要實機 Cocos Creator + 完整 API 知識，當前無此基礎。
- 漸進式 refactor 能持續產出可用 release，重寫期間 fork 將處於不可用狀態。

### Option C: 維持上游風格、僅修硬傷不重構

**為什麼沒選**：

- 自寫 HTTP server 與 JSON Schema 的維護成本長期累積，且 MCP 生態已標準
  化（`@modelcontextprotocol/sdk` 穩定）。
- 不換 SDK 就拿不到 Resources/Prompts/Notifications，AI 互動體驗會落後。
- 「只修硬傷」會在某個臨界點還是要重構，延遲只是把成本後推。

## 補註：P2 工具收斂量測結果（2026-05-01）

ADR 原本把「工具收斂」留作條件式決定（見 roadmap/03）。本日跑了量測 script
`scripts/measure-tool-tokens.js`，結論：**P2 不值得執行，已 close**。

### 量測方法

- 載入 dist 真實 registry（用 stub Editor 繞過 Cocos 依賴），抓全 160 個
  tool 的 schema，組成 MCP `tools/list` payload。
- 模擬 P2 兩種收斂形態：
  - **router-A（lossless）**：14 個 category 各包成一個 action router，
    用 `oneOf` discriminator 保留每個 sub-schema → 對外 arg validation
    與現況等價。這是 P2 真要 ship 的形態。
  - **router-B（lossy）**：14 個 router 只保留 action enum + 自由 `args:
    object`，丟掉 per-action schema。這是 v1.5.0 「-50% tokens」行銷數字
    唯一能達成的形態，代價是失去 schema-driven validation。
- 用 chars / 3.5 估 token（JSON-heavy 內容的 Claude tokenizer 中位數），
  另算 ratios 3.0 / 4.0 作上下界。決策門檻 30% / 15% 寬到 ratio 精度
  不影響結論。
- 對 5 個樣本 tool 經由真實 SDK server 跑 round-trip，量 request +
  response bytes。

### 結果（chars，tokens 以 mid ratio 估）

| 形態 | chars | tokens (mid) | vs current |
|---|---:|---:|---:|
| current（160 flat） | 51,983 | ~14,852 | — |
| router-A（lossless oneOf） | 71,278 | ~20,365 | **+37.1%（更大）** |
| router-B（lossy enum-only） | 17,143 | ~4,898 | -67.0% |

逐 category 14 組裡，**router-A 全部都比 flat 大**（最低 -13%、最高
-99%），原因是 oneOf 每個分支都重複帶 `title` / `description` /
`properties` / `required` 框架，加上每個 sub-schema 多一層 `args:`
wrapper，省下的「tool name / category prefix 重複」抵不過框架成本。

Round-trip：5 個樣本平均 ~257 tokens / call。P2 形態加一個 `action`
field，每 call 多 ~6 tokens，在 round-trip 中佔比 < 3%，可忽略。

### 結論

- **lossless P2 反而讓 prompt 多 37%**——是負收益，不是降。
- **lossy P2** 雖然能砍 67%，但等於把 v1.5.0 的「降 token」承諾建立在
  丟掉 arg validation 上。對 Cocos Creator 這種「每個 tool 參數差異大、
  resource UUID / dump path 容易打錯」的領域，丟 validation 是退步。
- 上游 v1.5.0 README 宣稱的 -50% token 因此被證實為**只在 lossy 形態
  下成立的行銷數字**，不是無代價的優化。
- P2 整段 close（roadmap/03 已標 Closed，附本量測連結）。後續若要再降
  prompt 預算，方向應該是 (a) Resources/Prompts MCP feature 把長期不變
  的 schema 移出 prompt，或 (b) 進 ToolManager 砍掉冷門 tool 集合。

ADR 0001 結論不變：不追 v1.5.0 spec、自走路線。本量測是把「工具收斂」
這條附帶決定變成已關閉狀態，不是改變主決策。
